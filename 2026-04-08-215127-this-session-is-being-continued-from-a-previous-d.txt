⏺ With chunk-level embeddings in place, the big unlocks are:

  1. "Ask your library" chat — RAG-powered Q&A where users ask natural language
  questions and get answers citing specific passages from their resources. This
  is the killer feature for a PKMS.
  2. Passage-level semantic search — Instead of just matching resource titles,
  search returns the specific paragraph that answers the query, with the source
  resource linked.
  3. Auto-generated highlights — When viewing a resource, surface the most
  relevant passages from other resources in the library (cross-resource passage
  linking).
  4. Smart daily digest — Periodic summary of connections between recently added
   resources ("Your article on X relates to this passage from Y you saved last
  week").
  5. Collection auto-organization — Suggest which collection a new resource
  belongs in based on chunk similarity to existing collection contents.

  The highest-impact, most differentiating feature is #1 — "Ask your library".
  It turns a passive bookmark/note store into an active knowledge assistant.
  Want to start on that?

⏺ Yeah essentially — a chat interface where the context is your library. User

❯ what about mcp stuff in the ai chat?                                          

⏺ MCP (Model Context Protocol) would let users connect external tools/data
  sources to the chat — like pulling in data from Notion, Linear, Slack, Google
  Drive, etc. without us building each integration ourselves.

  But that's a big scope expansion and MCP is still early. I'd recommend:

  Phase 1 — Build the chat with our own internal tools (search library, cite
  passages, create notes, tag resources). This is the core value.

  Phase 2 — Add MCP client support so users can plug in external MCP servers.
  The chat becomes a universal knowledge interface, not just for what's saved in
   Strand.

  Phase 1 is what makes the product useful. Phase 2 is what makes it a platform.
   Let's start with Phase 1?
